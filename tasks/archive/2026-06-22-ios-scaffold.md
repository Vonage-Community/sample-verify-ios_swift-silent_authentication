## Task 5 — iOS project scaffold

### Goal

Create a compilable, testable Xcode project for SilentAuthDemo with the correct structure, Swift Package Manager dependency on `VonageClientLibrary`, and iPhone + iPad support. No application logic yet — just the skeleton that later tasks will fill in.

### Constraints discovered during planning

- **Xcode is not installed** on this machine (only Swift CLI + `xcodeproj` Ruby gem). The `.xcodeproj` bundle will be generated by a Ruby script using the `xcodeproj` gem, then verified with `xcodebuild` (which requires the full Xcode app to be installed on the user's machine before running the verification step).
- **xcodeproj gem** (`v1.27.0`) is installed at `$(ruby -e 'puts Gem.user_dir')/bin/xcodeproj`. SPM dependencies cannot be added via the gem alone — SPM is stored in `project.pbxproj` as `XCRemoteSwiftPackageReference` entries, which we will write directly into the generated pbxproj.
- **Verification command** requires Xcode to be installed on the machine running it. The plan flags this explicitly — the scaffold can be created without Xcode, but `xcodebuild test` must be run by the user (or a CI machine) that has Xcode installed.

### Decisions

- **Generate `.xcodeproj` via Ruby script** rather than creating `project.pbxproj` by hand. The `xcodeproj` gem produces a well-formed pbxproj; adding the SPM remote reference is done via the gem's `project.add_swift_package` API. The script is committed to `scripts/create_xcodeproj.rb` so it's reproducible.
- **Test target from the start.** Per AGENTS.md, tests come before or alongside code. The Xcode project will include a `SilentAuthDemoTests` target from day one, wired to the app target. An initial `SmokeTests.swift` test (verifying the bundle identifier) will be written before the project is considered done, serving as the "does the build even work" canary.
- **`Config.xcconfig`** for build settings (base URL). The script applies it to both Debug and Release configurations.
- **Folder structure created as real filesystem directories** before the Xcode project, so the generator can reference them. Files are added as group references (not folder references) so Xcode can manage them individually.

## Blog-worthy notes

**Generating Xcode projects programmatically:** The `.xcodeproj` bundle is generated via `xcodeproj` Ruby gem (committed as `scripts/create_xcodeproj.rb`) rather than created manually in Xcode. This makes the project generation reproducible and auditable — the script is the source of truth. The pbxproj file is commit-safe; git conflicts in generated files are rare because the structure is always deterministic. Downside: the Ruby gem API is less documented than Xcode's UI, so discovering the right method calls (e.g., `XCRemoteSwiftPackageReference` construction, `PBXBuildFile.product_ref` for SPM products) requires exploring the gem's source.

**Swift Package Manager resolution on first build:** SPM resolves package dependencies on first build (`VonageClientLibrary 1.0.5` in this case), not at project generation time. The `.xcodeproj` only contains the remote reference; actual resolution happens when `xcodebuild` first runs. This is good (network happens at build time, not project generation), but means the exact resolved version isn't captured in the repo until someone runs a build and commits the updated `Package.resolved` file if you were using a Package.swift-based project (we're not — this is just noting the behavior for future reference).

**Test target boilerplate:** iOS test bundles linked to an app target require either an explicit `Info.plist` or `GENERATE_INFOPLIST_FILE = YES` in build settings. The generator uses `GENERATE_INFOPLIST_FILE` so the test bundle doesn't need a separate plist file, keeping the file count down. The test target is wired to the app target via `TEST_HOST` and `BUNDLE_LOADER` build settings — those two lines establish the "unit test of an app" relationship in Xcode.

**File reference paths in xcodeproj:** A subtle pbxproj gotcha: when creating file references via the gem, the path is relative to the containing group, not the project root. Passing a full path like `File.join('SilentAuthDemo', 'App', 'SilentAuthDemoApp.swift')` to `group.new_file()` when the group is already `App/` results in double-nesting (`SilentAuthDemo/App/SilentAuthDemo/App/SilentAuthDemoApp.swift`). The fix: pass just the filename to `new_file()`, and let the group hierarchy provide the directory structure.
